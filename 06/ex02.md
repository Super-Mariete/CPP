# Ex02 — Identify real type, explicado desde cero

## ¿Qué hace el programa?

Lo ejecutas y te imprime algo así (aleatorio cada vez):

```
Via pointer : A
Via reference: A

Via pointer : C
Via reference: C

Via pointer : B
Via reference: B
```

Crea objetos de tipo `A`, `B` o `C` de forma aleatoria, pero los "envuelve" en un puntero de tipo `Base*`. El reto es descubrir cuál de los tres es en realidad, sin que el código sepa de antemano cuál va a llegar.

---

## Concepto 1: herencia — la familia de clases

```cpp
class Base
{
public:
    virtual ~Base();
};

class A : public Base {};
class B : public Base {};
class C : public Base {};
```

La **herencia** es uno de los pilares de la programación orientada a objetos. Significa que una clase puede "ser hija" de otra y heredar su comportamiento.

Piénsalo como una familia:
- `Base` es el padre
- `A`, `B` y `C` son hijos

`class A : public Base {}` se lee: "la clase `A` hereda públicamente de `Base`". El `public` aquí significa que todo lo que era público en `Base` sigue siendo público en `A`.

Consecuencia importante: donde el código espera un `Base*`, puedes pasar un `A*`, un `B*` o un `C*`. Un hijo siempre puede hacerse pasar por su padre. Esto se llama **polimorfismo**.

---

## Concepto 2: el destructor virtual — por qué es obligatorio

```cpp
class Base
{
public:
    virtual ~Base();
};
```

Un **destructor** es la función que se llama cuando un objeto se destruye (cuando su memoria se libera). Se escribe con `~` antes del nombre de la clase.

¿Por qué `virtual`? Porque sin `virtual`, cuando haces:

```cpp
Base* obj = new A();
delete obj;   // ← sin virtual, llama al destructor de Base, no al de A
```

El programa llama al destructor de `Base` (el tipo del puntero), no al destructor de `A` (el tipo real). Si `A` tuviera recursos propios que limpiar, se quedarían sin limpiar — **memory leak**.

Con `virtual ~Base()`, el programa busca el destructor correcto en tiempo de ejecución y llama al de `A`. Eso es polimorfismo aplicado a la destrucción.

**Pero hay un segundo motivo, más importante para este ejercicio**: sin `virtual` en la clase base, no hay **vtable**, y sin vtable no hay **RTTI** (Run-Time Type Information). `dynamic_cast` necesita RTTI para funcionar. Si `Base` no tiene ninguna función virtual, `dynamic_cast` directamente no compila sobre ella.

---

## Concepto 3: vtable y RTTI — cómo el programa sabe el tipo real

Cuando una clase tiene al menos una función `virtual`, el compilador crea una estructura interna llamada **vtable** (tabla de funciones virtuales). Es una tabla que cada objeto lleva consigo con punteros a sus funciones reales.

```
objeto A en memoria:
┌─────────────────┐
│ puntero a vtable │ ← apunta a la vtable de A
│ (datos de A)    │
└─────────────────┘

vtable de A:
┌───────────────────────────┐
│ puntero a ~A()            │
│ información de tipo: "A"  │  ← esto es el RTTI
└───────────────────────────┘
```

El **RTTI** (Run-Time Type Information) es esa "información de tipo" guardada en la vtable. Es lo que permite preguntar en tiempo de ejecución: "¿este objeto que tengo como `Base*`, qué tipo real tiene?".

Sin `virtual`, no hay vtable, no hay RTTI, y no hay forma de responder esa pregunta.

---

## Concepto 4: `new` y `delete` — memoria dinámica

```cpp
return new A();
```

`new` crea un objeto en el **heap** (montón), la zona de memoria que se gestiona manualmente. A diferencia de las variables normales (que viven en el stack y se destruyen solas), un objeto creado con `new` vive hasta que tú mismo llames a `delete`.

```cpp
delete obj;
```

`delete` llama al destructor del objeto y libera la memoria. Si no llamas a `delete`, la memoria queda ocupada hasta que el programa termina — eso es un **memory leak** (fuga de memoria).

---

## Concepto 5: `generate()` — creación aleatoria

```cpp
Base* generate()
{
    switch (std::rand() % 3)
    {
        case 0: return new A();
        case 1: return new B();
        default: return new C();
    }
}
```

`std::rand()` genera un número entero pseudoaleatorio. El operador `%` es el módulo (resto de la división entera), así que `rand() % 3` da siempre 0, 1 o 2.

El resultado es un `Base*` que apunta a un `A`, `B` o `C` según el número aleatorio. El código que llama a `generate()` recibe un `Base*` y **no sabe** cuál de los tres es realmente. Eso es exactamente el problema que hay que resolver.

En el `main`, antes del bucle:

```cpp
std::srand(std::time(NULL));
```

`srand` inicializa la semilla del generador aleatorio. Si no lo haces, `rand()` siempre da la misma secuencia. `time(NULL)` devuelve el tiempo actual en segundos — cambia cada segundo, así que la secuencia es diferente cada vez que ejecutas el programa.

---

## Concepto 6: `dynamic_cast` — el cast del ejercicio

`dynamic_cast` es el único cast de C++ que opera en **tiempo de ejecución** (runtime) consultando el RTTI. Intenta convertir un puntero o referencia a un tipo más específico dentro de una jerarquía. Si el objeto real no es de ese tipo, falla.

El comportamiento del fallo depende de si usas puntero o referencia — y esa diferencia es la clave del ejercicio.

---

## Concepto 7: `identify(Base* p)` — fallo con puntero nulo

```cpp
void identify(Base* p)
{
    if (dynamic_cast<A*>(p))
        std::cout << "A" << std::endl;
    else if (dynamic_cast<B*>(p))
        std::cout << "B" << std::endl;
    else if (dynamic_cast<C*>(p))
        std::cout << "C" << std::endl;
    else
        std::cout << "Unknown" << std::endl;
}
```

`dynamic_cast<A*>(p)` intenta tratar `p` como un puntero a `A`. Consulta el RTTI del objeto al que apunta `p`:

- Si el objeto real **es** un `A` → devuelve un `A*` válido (no nulo) → el `if` es verdadero
- Si el objeto real **no es** un `A` → devuelve `nullptr` → el `if` es falso

`nullptr` es el puntero nulo en C++: un puntero que no apunta a ningún sitio válido. En un `if`, se evalúa como `false`. Cualquier puntero no nulo se evalúa como `true`.

Por eso los `if` funcionan directamente: no hace falta comparar explícitamente con `nullptr`. El cast fallido devuelve nulo, el `if` lo detecta como falso, y pasa al siguiente.

---

## Concepto 8: `identify(Base& p)` — fallo con excepción

```cpp
void identify(Base& p)
{
    try {
        (void)dynamic_cast<A&>(p);
        std::cout << "A" << std::endl;
        return;
    } catch (const std::exception&) {}

    try {
        (void)dynamic_cast<B&>(p);
        std::cout << "B" << std::endl;
        return;
    } catch (const std::exception&) {}

    try {
        (void)dynamic_cast<C&>(p);
        std::cout << "C" << std::endl;
        return;
    } catch (const std::exception&) {}

    std::cout << "Unknown" << std::endl;
}
```

Aquí el parámetro es `Base&` — una **referencia** a `Base`, no un puntero.

¿Cuál es la diferencia? Una referencia es un alias de un objeto existente. No puede ser nula (en C++ no existe la "referencia nula"). Por eso `dynamic_cast` sobre una referencia no puede devolver "nada" cuando falla — no tiene nada válido que devolver. En su lugar, **lanza una excepción** de tipo `std::bad_cast`.

**¿Qué es una excepción?** Es un mecanismo para señalar que algo salió mal. Cuando el programa "lanza" (`throw`) una excepción, la ejecución normal se interrumpe y sube por la pila de llamadas hasta encontrar un bloque `try/catch` que la atrape.

```
try { ... }         ← "intenta ejecutar esto"
catch (tipo& e) {}  ← "si lanza una excepción de este tipo, atrápala aquí"
```

El flujo en este código cuando el objeto es un `B`:

1. `dynamic_cast<A&>(p)` → falla → lanza `std::bad_cast`
2. El `catch` lo atrapa, el bloque `try` termina sin imprimir nada
3. `dynamic_cast<B&>(p)` → tiene éxito → no lanza nada
4. Se imprime `"B"` y se hace `return`

`(void)dynamic_cast<A&>(p)` — el `(void)` es para decirle al compilador que el resultado del cast no se usa intencionalmente (evita una advertencia). Lo que importa es si lanza o no lanza.

**¿Por qué no usar punteros dentro de `identify(Base& p)`?** El subject lo prohíbe explícitamente. La idea es forzar a entender que con referencias el mecanismo de fallo es diferente. No hay "referencia nula" que comprobar, solo excepciones que capturar.

---

## Concepto 9: sobrecarga de funciones

```cpp
void identify(Base* p);   // versión con puntero
void identify(Base& p);   // versión con referencia
```

Dos funciones con el mismo nombre pero distintos parámetros. Esto se llama **sobrecarga** (overloading). El compilador decide cuál llamar según el tipo del argumento que le pasas.

En el `main`:

```cpp
identify(obj);    // obj es Base* → llama a identify(Base*)
identify(*obj);   // *obj desreferencia el puntero → es un Base& → llama a identify(Base&)
```

`*obj` es el operador de **desreferencia**: si `obj` es un puntero, `*obj` es el objeto al que apunta. Al pasarlo a una función que espera `Base&`, automáticamente se crea la referencia.

---

## Concepto 10: `<typeinfo>` está prohibido — por qué

El subject prohíbe incluir `<typeinfo>`. Ese header da acceso a `typeid()`, que devuelve el tipo real de un objeto directamente:

```cpp
typeid(*obj).name()   // daría "A", "B" o "C" sin esfuerzo
```

Si se permitiera, el ejercicio no tendría ningún mérito. El objetivo es aprender a usar `dynamic_cast` para identificar tipos en jerarquías polimórficas. `typeid` cortocircuita ese aprendizaje.

---

## El flujo completo del `main`

```cpp
std::srand(std::time(NULL));   // semilla aleatoria basada en el tiempo

for (int i = 0; i < 6; i++)
{
    Base* obj = generate();    // crea A, B o C aleatoriamente, devuelve Base*

    std::cout << "Via pointer : ";
    identify(obj);             // dynamic_cast con puntero, fallo → nullptr

    std::cout << "Via reference: ";
    identify(*obj);            // dynamic_cast con referencia, fallo → excepción

    std::cout << std::endl;
    delete obj;                // libera la memoria del heap
}
```

Seis iteraciones, cada una con un objeto distinto creado aleatoriamente. Después de cada par de identificaciones, `delete` libera la memoria. Sin ese `delete` habría un memory leak de 6 objetos.

---

## Por qué hay dos versiones de `identify`

No es capricho. Refleja una diferencia real del lenguaje:

| | Con puntero | Con referencia |
|---|---|---|
| Puede ser nulo | Sí (`nullptr`) | No (las referencias no pueden ser nulas) |
| Cast falla | Devuelve `nullptr` | Lanza `std::bad_cast` |
| Cómo detectar | Comprobar si es `nullptr` en un `if` | Bloque `try/catch` |

Cuando recibes un `Base*`, hay una posibilidad de que sea `nullptr` (puntero nulo). El `dynamic_cast` puede explotar esa posibilidad para señalar el fallo silenciosamente.

Cuando recibes un `Base&`, el objeto ya existe con certeza (las referencias no pueden referirse a nada). No hay "nada" que devolver como señal de fallo, así que el mecanismo es la excepción.

---

## Resumen visual

```
generate() crea aleatoriamente uno de:
   new A()  ──┐
   new B()  ──┤──→ devuelve Base*
   new C()  ──┘        ↓
                   obj (Base*)
                    ↙        ↘
          identify(obj)    identify(*obj)
         (Base* versión)   (Base& versión)
               ↓                 ↓
       dynamic_cast<A*>   dynamic_cast<A&>
       fallo → nullptr    fallo → std::bad_cast
       éxito → A*         éxito → continúa sin lanzar
               ↓                 ↓
          imprime "A"        imprime "A"

Al final: delete obj  ← libera memoria del heap
```

El tipo real del objeto siempre fue `A`, `B` o `C`. El tipo aparente era `Base*`. `dynamic_cast` es el único mecanismo de C++ que puede mirar "debajo" del tipo aparente y revelar el tipo real, consultando el RTTI guardado en la vtable del objeto.

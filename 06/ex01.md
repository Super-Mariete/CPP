# Ex01 — Serializer, explicado desde cero

## ¿Qué hace el programa?

Lo ejecutas y te imprime algo así:

```
Original pointer : 0x7ffd3a2b1c10
name  : Marvin
value : 42

Serialized (uintptr_t): 140726184099856

Deserialized pointer : 0x7ffd3a2b1c10
name  : Marvin
value : 42

Pointers match: yes
```

Coge un objeto en memoria, lo convierte a un número entero (`serialize`), y luego vuelve a convertir ese número al objeto original (`deserialize`). Al final comprueba que el puntero recuperado es exactamente el mismo que el original.

---

## Concepto 1: la memoria del ordenador

Imagina la memoria RAM como una calle muy larga con casas numeradas. Cada casa es un byte de memoria y tiene un número de casa (su **dirección**).

Cuando creas un objeto en un programa:

```cpp
Data data;
data.name = "Marvin";
data.value = 42;
```

El sistema operativo te asigna unas casas de esa calle para guardar los datos de `data`. Por ejemplo, a partir de la casa número `140726184099856`. Eso es la **dirección de memoria** del objeto.

---

## Concepto 2: `struct` — una agrupación de datos

```cpp
struct Data
{
    std::string name;
    int         value;
};
```

Una `struct` es una forma de agrupar varios datos bajo un mismo nombre. Es como un formulario con campos. Aquí `Data` tiene dos campos: `name` (texto) y `value` (número entero).

La diferencia entre `struct` y `class` en C++ es casi solo de convenciones: en `struct` todo es público por defecto, en `class` todo es privado por defecto. En este ejercicio se usa `struct` porque es un contenedor simple de datos, sin comportamiento.

---

## Concepto 3: punteros — la dirección de un objeto

```cpp
Data* ptr = &data;
```

- `&data` — el operador `&` devuelve la **dirección** de `data` en memoria (el número de casa).
- `Data*` — un **puntero** a `Data`. Una variable que guarda una dirección de memoria (no el objeto en sí, sino dónde está).

Diferencia clave:

```cpp
Data  data;    // el objeto en sí (las casas con los muebles dentro)
Data* ptr;     // un papel donde apuntas el número de la casa
```

Para acceder a los datos a través de un puntero se usa `->`:

```cpp
ptr->name;    // equivale a: (*ptr).name
```

El `->` significa "ve a la dirección que tengo apuntada y dame ese campo".

---

## Concepto 4: `uintptr_t` — un entero del tamaño justo para una dirección

```cpp
#include <stdint.h>

uintptr_t raw = Serializer::serialize(&data);
```

`uintptr_t` es un tipo entero especial garantizado de ser lo suficientemente grande para guardar una dirección de memoria. En un sistema de 64 bits ocupa 8 bytes; en uno de 32 bits ocupa 4 bytes.

¿Por qué no usar simplemente `int`? Porque `int` normalmente tiene 4 bytes, y en sistemas modernos de 64 bits las direcciones necesitan 8 bytes. Si usaras `int`, los números más grandes no cabrían y perderías información. `uintptr_t` es la promesa del estándar de que siempre cabe una dirección, sea cual sea la plataforma.

El prefijo `u` significa "unsigned" (sin signo, solo positivos). Las direcciones de memoria nunca son negativas.

---

## Concepto 5: la clase `Serializer` — misma estructura que ex00

```cpp
class Serializer
{
public:
    static uintptr_t serialize(Data* ptr);
    static Data*     deserialize(uintptr_t raw);

private:
    Serializer();
    Serializer(const Serializer&);
    Serializer& operator=(const Serializer&);
    ~Serializer();
};
```

Idéntico patrón que `ScalarConverter`: una **utility class** que no se puede instanciar. Los constructores están en `private`, así que nadie puede hacer `Serializer s;`. Las dos funciones son `static`, lo que significa que se llaman directamente sobre la clase:

```cpp
Serializer::serialize(&data);      // sin crear ningún objeto
Serializer::deserialize(raw);
```

## Concepto 6: `reinterpret_cast` — el cast del ejercicio

```cpp
uintptr_t Serializer::serialize(Data* ptr)
{
    return reinterpret_cast<uintptr_t>(ptr);
}

Data* Serializer::deserialize(uintptr_t raw)
{
    return reinterpret_cast<Data*>(raw);
}
```

Aquí está el núcleo del ejercicio. `reinterpret_cast` es el cast más "peligroso" y directo de C++. No convierte el valor — **reinterpreta los mismos bytes como si fueran otro tipo**.

Comparación con `static_cast` (el del ex00):

| | `static_cast` | `reinterpret_cast` |
|---|---|---|
| Qué hace | Convierte el valor (puede cambiar los bits) | Reinterpreta los mismos bits sin tocarlos |
| Seguridad | Verificado en compilación | Sin garantías, tú asumes la responsabilidad |
| Ejemplo | `int` 65 → `double` 65.0 (bits distintos) | puntero `0x7ffd...` → entero `140726...` (mismos bits) |

Concretamente:

```cpp
reinterpret_cast<uintptr_t>(ptr)
```

Coge los 8 bytes del puntero `ptr` y los lee como si fueran un entero sin signo. El número resultante es exactamente la dirección de memoria del objeto.

```cpp
reinterpret_cast<Data*>(raw)
```

Coge los 8 bytes del entero `raw` y los lee como si fueran un puntero a `Data`. Si ese número era una dirección válida (y aquí lo es, porque nadie liberó la memoria), el puntero resultante apunta exactamente al mismo objeto.

¿Por qué no funciona `static_cast` aquí? Porque `static_cast` no permite convertir entre punteros y enteros — son tipos demasiado distintos. El compilador lo rechaza. Solo `reinterpret_cast` lo permite, porque su contrato es "reinterpreta los bits sin preguntar".

---

## Concepto 7: el `#ifndef` — guardia de inclusión

```cpp
#ifndef SERIALIZER_HPP
# define SERIALIZER_HPP

// ... contenido del fichero ...

#endif
```

Esto aparece en todos los `.hpp` y sirve para evitar que el mismo fichero se incluya dos veces. Si el compilador ya procesó `SERIALIZER_HPP`, la segunda vez que encuentre `#include "Serializer.hpp"` salta todo el contenido. Sin esto, el compilador definiría la clase dos veces y daría error.

## El flujo completo del `main`

```cpp
Data data;
data.name = "Marvin";
data.value = 42;
```

Se crea un objeto `Data` en la pila (stack) con dos campos.

```cpp
uintptr_t raw = Serializer::serialize(&data);
```

- `&data` — obtiene la dirección de `data` (un puntero `Data*`)
- `serialize` — convierte ese puntero a un entero con `reinterpret_cast`
- `raw` guarda ese número (la dirección de memoria como entero)

```cpp
Data* recovered = Serializer::deserialize(raw);
```

- `deserialize` — convierte el entero de vuelta a un puntero con `reinterpret_cast`
- `recovered` apunta al mismo sitio en memoria que `data`

```cpp
std::cout << "\nPointers match: " << (&data == recovered ? "yes" : "no") << std::endl;
```

Compara las dos direcciones. Son iguales porque nadie liberó ni movió la memoria entre serialize y deserialize.

El operador `? :` es el **operador ternario**: `condición ? valor_si_true : valor_si_false`. Es un if-else compacto para usar dentro de una expresión.

## Por qué este ejercicio es importante

Demuestra que **un puntero no es magia**: es un número entero que representa una dirección. Puedes sacarlo del sistema de tipos, guardarlo como entero (en una base de datos, enviarlo por red, almacenarlo en disco), y si la memoria sigue siendo válida cuando lo recuperas, funciona exactamente igual.

Esto es lo que hace la serialización binaria real: convertir objetos en memoria a bytes para guardarlos o enviarlos, y luego reconstruirlos.

## Resumen visual

```
data (objeto en memoria, en la dirección 0x7ffd...)
        ↓  &data → puntero Data*
   serialize()
        ↓  reinterpret_cast<uintptr_t>
  raw = 140726184099856  (el mismo número, leído como entero)
        ↓
  deserialize()
        ↓  reinterpret_cast<Data*>
  recovered (puntero Data*, apunta a la misma dirección)
        ↓
  recovered->name  == "Marvin"   ✓
  recovered->value == 42         ✓
  &data == recovered             ✓
```

El objeto no se mueve, no se copia. Solo se cambia la forma de ver su dirección: primero como puntero, luego como entero, luego como puntero otra vez.

## UTILITY CLASS
Una utility class es una clase que existe solo para agrupar funciones relacionadas. No representa ningún objeto del mundo real, no guarda   
  datos, no tiene estado.                                                                                                                   
                                                                                                                                              
  ---
  Sin utility class — funciones sueltas                                                                                                       
                                                                                                                                              
  // funciones sueltas en el archivo, sin clase
  uintptr_t serialize(Data* ptr);                                                                                                             
  Data*      deserialize(uintptr_t raw);
                                                                                                                                              
  Funciona, pero las funciones flotan en el espacio global. Si el proyecto crece, puedes tener colisiones de nombres con otras funciones de   
  otras partes del código.                                                                                                                    
                                                                                                                                              
  ---             
  Con utility class

  class Serializer
  {
  public:
      static uintptr_t serialize(Data* ptr);
      static Data*     deserialize(uintptr_t raw);                                                                                            
   
  private:                                                                                                                                    
      Serializer();   // prohibido crear objetos
  };                                                                                                                                          
   
  Las funciones viven dentro del namespace de la clase. Las llamas así:                                                                       
                  
  Serializer::serialize(&data);    // queda claro de dónde viene                                                                              
                  
  El Serializer:: actúa como carpeta. En un proyecto grande puedes tener Serializer::serialize, Compressor::serialize, Encryptor::serialize — 
  sin colisiones. 
                                                                                                                                              
  ---             
  ¿Por qué prohibir el constructor?
                                                                                                                                              
  Porque si dejas el constructor público, alguien podría hacer:
                                                                                                                                              
  Serializer s;          // crear un objeto que no sirve para nada
  s.serialize(&data);    // funciona igual, pero es confuso                                                                                   
                                                                                                                                              
  Al poner los constructores en private el compilador da error si alguien intenta instanciarla. Le estás comunicando al que lea el código:    
  esta clase no es para crear objetos, es solo una caja de herramientas.                                                                      
                                                                                                                                              
  ---             
  La diferencia con una clase normal
                                    
  ┌──────────────┬────────────────────────────┬─────────────────────────────────┐
  │              │        Clase normal        │          Utility class          │                                                             
  ├──────────────┼────────────────────────────┼─────────────────────────────────┤
  │ Representa   │ Un objeto (Coche, Usuario) │ Una categoría de funciones      │                                                             
  ├──────────────┼────────────────────────────┼─────────────────────────────────┤
  │ Tiene datos  │ Sí (_velocidad, _nombre)   │ No                              │                                                             
  ├──────────────┼────────────────────────────┼─────────────────────────────────┤                                                             
  │ Se instancia │ Sí (Coche c;)              │ No (constructor privado)        │                                                             
  ├──────────────┼────────────────────────────┼─────────────────────────────────┤                                                             
  │ Sus métodos  │ Usan this                  │ Son static, no necesitan objeto │
  ├──────────────┼────────────────────────────┼─────────────────────────────────┤                                                             
  │ Ejemplo real │ std::string                │ std::sqrt, ScalarConverter      │
  └──────────────┴────────────────────────────┴─────────────────────────────────┘                                                             
                  
  ---                                                                                                                                         
  La pregunta que te puede hacer el evaluador
                                             
  "¿Por qué no usaste funciones libres directamente?"
                                                                                                                                              
  ▎ "Podría haberlo hecho. La diferencia es que la utility class agrupa las funciones bajo un nombre y evita contaminar el espacio de nombres 
  ▎ global. Es una convención de diseño que comunica intención: estas dos funciones van siempre juntas y pertenecen al mismo concepto."

  Ex02 — Array.hpp: clase template que se comporta como un array, explicada desde cero


  ¿Qué hace el ejercicio?

  Tienes que crear una clase template Array<T> que se comporte como un array
  pero con comprobación de acceso fuera de rango. La usas así:

  Array<int> ints(5);      // array de 5 enteros
  ints[0] = 42;            // asignar valor
  std::cout << ints[0];   // leer valor
  ints[10];                // lanza una excepción (fuera de rango)

  A diferencia de los arrays de C, si accedes a una posición que no existe,
  el programa no se rompe silenciosamente: lanza un error que puedes capturar.

  ---
  Concepto 1: qué es una clase template

  Ya viste funciones template (swap, min, max, iter). Una clase template es
  lo mismo pero para una clase entera: el tipo T es el tipo de los elementos
  que guarda.

  template <typename T>
  class Array
  {
      // aquí T puede ser int, string, Awesome, lo que sea
  };

  Cuando creas un objeto de la clase, debes especificar el tipo:

  Array<int>         ints(5);    // T = int
  Array<std::string> words(3);   // T = std::string
  Array<double>      nums(10);   // T = double

  A diferencia de las funciones template, el compilador NO puede deducir T
  automáticamente para clases. Tienes que escribirlo siempre.

  ---
  Concepto 2: memoria dinámica — qué es el heap

  En C++, la memoria se divide en dos zonas principales:

  ┌────────────────────────────────────────────────────────────────┐
  │  STACK (pila)  →  variables locales, se liberan solos         │
  │  HEAP (montón) →  memoria reservada con new, hay que liberar  │
  └────────────────────────────────────────────────────────────────┘

  Cuando escribes:
  int arr[5];              // stack: se destruye cuando acaba la función
  int* arr = new int[5];   // heap: existe hasta que tú hagas delete[]

  Array necesita vivir más allá de la función que la crea, y su tamaño
  se conoce en tiempo de ejecución (no en compilación). Por eso usa el heap.

  El ejercicio exige explícitamente usar new[]. Si usas un array estático
  de tamaño fijo, no cuenta.

  ---
  Concepto 3: los miembros privados

  private:
      T*          _data;    // puntero al array en el heap
      std::size_t _size;    // número de elementos

  _data es un puntero al primer elemento de la memoria reservada en el heap.
  _size guarda cuántos elementos hay.

  El guión bajo _ delante es una convención de estilo que indica que son
  atributos privados de la clase.

  ---
  Concepto 4: los constructores — cómo se crean los objetos

  CONSTRUCTOR VACÍO (sin parámetros):

  Array() : _data(new T[0]()), _size(0) {}

  La parte : _data(...), _size(0) es una lista de inicialización. Es la forma
  correcta en C++ de inicializar los miembros en el constructor.

  new T[0]() reserva un array de cero elementos. Puede parecer raro, pero es
  válido en C++: devuelve un puntero válido. _size queda en 0.

  CONSTRUCTOR CON TAMAÑO:

  Array(unsigned int n) : _data(new T[n]()), _size(n) {}

  new T[n]() reserva espacio para n elementos de tipo T.
  Los paréntesis () al final inicializan todos los elementos:
  - Para int: a 0
  - Para double: a 0.0
  - Para clases: llama al constructor por defecto

  ┌─────────────────────────────────────────────────────────────┐
  │  new T[n]    →  elementos sin inicializar (basura posible)  │
  │  new T[n]()  →  elementos inicializados a valor por defecto │
  └─────────────────────────────────────────────────────────────┘

  ---
  Concepto 5: la Regla de los Tres (Rule of Three)

  Si tu clase gestiona recursos (memoria dinámica, archivos, conexiones...),
  necesitas definir obligatoriamente estos tres métodos:

  1. Destructor          →  libera la memoria al destruir el objeto
  2. Constructor de copia →  crea un objeto nuevo copiando los datos
  3. Operador =          →  asigna datos de un objeto a otro ya existente

  Si no los defines, el compilador los genera automáticamente. Pero los
  generados automáticamente hacen shallow copy (copia superficial), que
  con memoria dinámica causa problemas graves.

  ---
  Concepto 6: shallow copy vs deep copy — el problema central

  SHALLOW COPY (lo que haría el compilador por defecto):

  Array<int> a(5);
  a[0] = 42;

  Array<int> b = a;   // copia solo el puntero _data

  Ahora a._data y b._data apuntan a la MISMA memoria:

  ┌─────┐     ┌───────────────┐
  │ a   │────▶│ [42][0][0][0][0] │  ← misma memoria
  └─────┘     └───────────────┘
  ┌─────┐     ↗
  │ b   │────
  └─────┘

  Problema 1: si modificas b[0], también cambias a[0]. No son independientes.
  Problema 2: cuando se destruyen a y b, se llama delete[] dos veces sobre
              la misma memoria → comportamiento indefinido (crash o corrupción).

  DEEP COPY (lo que implementas tú):

  Array(const Array& other) : _data(new T[other._size]()), _size(other._size)
  {
      for (std::size_t i = 0; i < _size; i++)
          _data[i] = other._data[i];
  }

  Reservas NUEVA memoria y copias los datos uno a uno:

  ┌─────┐     ┌───────────────────┐
  │ a   │────▶│ [42][0][0][0][0]  │  ← memoria propia de a
  └─────┘     └───────────────────┘
  ┌─────┐     ┌───────────────────┐
  │ b   │────▶│ [42][0][0][0][0]  │  ← memoria propia de b (copia)
  └─────┘     └───────────────────┘

  Ahora son completamente independientes. Modificar una no afecta a la otra.

  ---
  Concepto 7: el destructor

  ~Array() { delete[] _data; }

  El destructor se llama automáticamente cuando el objeto deja de existir
  (sale del bloque where fue creado, o se hace delete en el heap).

  delete[] _data libera la memoria del array. El [] es obligatorio cuando
  reservaste con new[]. Usar delete sin [] es comportamiento indefinido.

  Sin el destructor, esa memoria nunca se liberaría → memory leak.

  ---
  Concepto 8: el operador de asignación

  Array& operator=(const Array& other)
  {
      if (this != &other)
      {
          delete[] _data;
          _size = other._size;
          _data = new T[_size]();
          for (std::size_t i = 0; i < _size; i++)
              _data[i] = other._data[i];
      }
      return *this;
  }

  La diferencia con el constructor de copia es que aquí el objeto ya existe
  y puede tener memoria propia que hay que liberar antes.

  ┌──────────────────────────────────────────────────────────────────────┐
  │  if (this != &other)  →  protección contra "arr = arr"              │
  │                           Sin esto harías delete[] _data y luego    │
  │                           intentarías copiar desde memoria borrada   │
  │                                                                      │
  │  delete[] _data       →  liberar la memoria anterior                │
  │  _data = new T[...]() →  reservar nueva memoria del tamaño correcto │
  │  copiar elementos     →  deep copy                                   │
  │  return *this         →  permite encadenar: a = b = c               │
  └──────────────────────────────────────────────────────────────────────┘

  this es un puntero especial que siempre apunta al objeto actual.
  *this es el objeto en sí. &other es la dirección de memory de other.

  ---
  Concepto 9: operator[] — acceder a los elementos

  T& operator[](std::size_t index)
  {
      if (index >= _size)
          throw std::out_of_range("Index out of bounds");
      return _data[index];
  }

  Dos cosas importantes:

  1. Devuelve T& (referencia), no T (copia). Esto permite:
     ints[0] = 42;   // asignar directamente al elemento
     Si devolviera T, asignarías a una copia temporal sin efecto.

  2. Lanza std::out_of_range si el índice está fuera del rango.
     index >= _size cubre tanto el caso de índice negativo (imposible con
     size_t que es unsigned) como el de índice demasiado grande.

  ---
  Concepto 10: las dos versiones de operator[]

  T& operator[](std::size_t index);             // versión normal
  const T& operator[](std::size_t index) const; // versión const

  ¿Por qué dos versiones?

  En C++ un método const solo puede llamarse sobre objetos const:

  Array<int> arr(5);         // objeto normal
  arr[0] = 42;               // usa la versión T& (permite escritura)

  const Array<int> carr(5);  // objeto const
  int x = carr[0];           // usa la versión const T& (solo lectura)
  carr[0] = 42;              // ERROR: no puedes escribir en un const

  Sin la versión const, si alguien tiene un const Array<T>, no podría
  ni siquiera leer sus elementos con []. La versión const existe para
  permitir la lectura manteniendo la garantía de no-modificación.

  ---
  Concepto 11: std::out_of_range y las excepciones

  throw std::out_of_range("Index out of bounds");

  throw interrumpe el flujo normal del programa y lanza una excepción.
  La excepción sube por la pila de llamadas hasta que alguien la capture.

  Para capturarla:

  try {
      std::cout << ints[10] << std::endl;    // índice inválido
  }
  catch (const std::exception& e) {
      std::cerr << "Exception: " << e.what() << std::endl;
  }

  std::out_of_range hereda de std::exception. e.what() devuelve el
  mensaje que escribiste al crear la excepción: "Index out of bounds".

  Si nadie captura la excepción, el programa termina con un error.

  ---
  Concepto 12: size() — consultar el tamaño

  unsigned int size() const { return static_cast<unsigned int>(_size); }

  El const al final significa que este método no modifica el objeto.
  Puedes llamarlo sobre objetos const.

  _size es std::size_t (sin signo). Se convierte a unsigned int con
  static_cast para devolver exactamente el tipo que pide el enunciado.

  ---
  Flujo completo: qué pasa en cada operación

  CREAR: Array<int> ints(5);
    → Constructor con n=5
    → _data = new int[5]()  →  reserva 5 ints, todos a 0
    → _size = 5

  ESCRIBIR: ints[0] = 42;
    → operator[](0): 0 < 5, ok
    → devuelve _data[0] como referencia
    → la asignación escribe 42 en _data[0]

  LEER: std::cout << ints[2];
    → operator[](2): 2 < 5, ok
    → devuelve _data[2] como referencia
    → std::cout lee el valor

  COPIAR: Array<int> copy(ints);
    → Constructor de copia
    → _data = new int[5]()  →  nueva memoria
    → copia _data[0..4] de ints
    → copy y ints son independientes

  FUERA DE RANGO: ints[10];
    → operator[](10): 10 >= 5
    → throw std::out_of_range("Index out of bounds")

  DESTRUIR: objeto sale de scope
    → ~Array(): delete[] _data  →  libera la memoria

  ---
  Resumen del diseño

  Array<T>
  ├── _data: T*          →  memoria en el heap (new T[n]())
  ├── _size: size_t      →  número de elementos
  │
  ├── Array()            →  array vacío (new T[0]())
  ├── Array(n)           →  array de n elementos inicializados
  ├── Array(other)       →  deep copy: nueva memoria + copia datos
  ├── operator=(other)   →  libera la propia + deep copy
  ├── ~Array()           →  delete[] _data
  │
  ├── operator[](i)      →  T&: lectura y escritura con comprobación
  ├── operator[](i) const →  const T&: solo lectura con comprobación
  └── size()             →  devuelve _size

  La clave del ejercicio es la deep copy (Rule of Three) y la gestión
  correcta de la memoria dinámica: new T[n]() en construcción,
  delete[] en destrucción, y nunca acceder a memoria no reservada.
